# 008 — 長時間空白導致 Smart Trimming 定位嚴重偏移

## 問題描述
當使用者在麥克風開啟後，先發呆或有長時間背景雜音（例如經過 80 秒），最後才說出目標句子（例如 "The resource... hypothesis"）。系統成功辨識到，但保存下來的音頻卻不包含該單字。

### 問題 Log
```
[AUDIO] ✂️ User buffer too long: 375 chunks (~96.0s)
[AUDIO] ✂️ Target "hypothesis" at word 11/19 (58%) → estimated audio pos: chunk 217/375
[AUDIO] ✂️ Trimming to chunks [187..247) = 60 chunks (~15.4s)
```

## 根本原因
**Audio Buffer 與 Transcript 的時間累積不同步（脫鉤）。**

1. **Audio Buffer** 忠實記錄了所有的時間（包含前面 80 秒的無對話空白）。總長度 96 秒。
2. **Transcript** 只記錄了有語音辨識出文字的片段，共有 19 個 fragments，目標字在第 11 個。
3. 我們的線性比例演算法：`58% * 96秒 = 第 55 秒`。
4. **錯誤發生**：實際上前面的 80 秒全都是空白。使用者真正的講話時間是第 81~96 秒！所以目標字真正的位子在第 90 秒附近（chunk 350）。
5. 我們的窗口截取了第 48~63 秒，完美切在了空白區。

## 解決方案修改思路

原始按照「字數比例 (idx/total)」乘上「總時間 (chunks)」的做法，**前提必須是語音充滿了整個時段**。只要前面有長空白，這個線性比例就會被徹底破壞。

### 新演算法：後端對齊 (Right-Aligned) 的動態窗口
既然我們是在即時語音中觸發 `MATCHED`，這意味著：**使用者「剛」講完這句話不久。** 目標字一定在 buffer 的「尾部區間」。

因此：
1. **設定合理的人類語句長度限制**：最高截取 100 chunks（約 25.6 秒）。這足夠涵蓋一般甚至超長的英文長句。
2. 不再用 `idx/total` 做全局線性投影，而是將截取範圍直接 **對齊到 buffer 尾端**。
3. 若目標字在 transcript 很前面的位置（例如 10%），我們就往前回溯多取一點；如果在很後面，就取靠近尾巴的片段，但一律保證以剛講完的結尾為基準點。
4. 最簡單且防呆的作法：如果超過 100 chunks (25s)，一律只取**最後的 100 chunks**。人類講包含一個單字的句子極少超過 25 秒，這能完全避開前面的發呆空白段。

### 結果對比

(情境：96 秒 buffer，單字剛唸完)
- **舊演算法（比例法）**：切在中間第 55 秒 → ❌
- **新演算法（尾部截斷法）**：如果大於 25 秒，直接取最後 25 秒 -> [第 71 秒 ~ 第 96 秒] → ✅ 穩穩涵蓋最近講話的片段。
